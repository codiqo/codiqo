## Caller Information Format

Each caller in the JSON includes:
- **callerMethod**: Caller name
- **file**: Source file path
- **line**: Line number of the call
- **isTestCaller**: Whether caller is in test code
- **signature**: Java ASM binary format signature (see below)
- **kind**: Symbol kind (constructor, function, lambda, etc.)
- **symbol**: Fully qualified symbol identifier
- **isDeprecated**: Whether caller is marked @Deprecated
- **callSiteCount**: Number of times this caller invokes the target (indicates coupling strength)

### ASM Signature Format

The `signature` field uses Java ASM binary descriptor format:
```
package/ClassName.methodName(parameterDescriptors)returnDescriptor
```

| Descriptor | Type |
|------------|------|
| `I` | int |
| `J` | long |
| `D` | double |
| `F` | float |
| `Z` | boolean |
| `B` | byte |
| `C` | char |
| `S` | short |
| `V` | void (return only) |
| `Lpackage/ClassName;` | object type |
| `[` | array prefix |

**Examples:**
- `com/example/Service.process(Ljava/lang/String;I)V` - void process(String, int)
- `com/example/Repo.findById(J)Lcom/example/Entity;` - Entity findById(long)
- `com/example/Util.transform([Ljava/lang/Object;)Ljava/util/List;` - List transform(Object[])

**IMPORTANT:** Cross-reference these sources. Metadata provides counts, but the diff and code blocks show what actually changed.

The user message includes a "PRE-COMPUTED VOLUME SCORES" section with:

## Volume Scores (Pre-Computed — Copy As-Is)

Volume is computed bottom-up from per-code-block efforts: `sum(cappedDriverScore × operationMult × testWeight)^[(${volume_exponent})] × filesScopeMultiplier`.
Per-block `driverScore` is a project-wide weighted average of three dimensions, each projected onto lines-equivalent units:
- `L` = body-only physical non-comment code lines of the method/constructor (signature/parameter lines are excluded)
- `S` = NCSS (logical statements)
- `I` = direct method invocations inside the block

Per code block, the request also reports `declarationCodeLines` — non-comment code lines occupied by the signature alone (modifiers, name, parameters, `throws` clause). It is `≥ 1` for any concrete or abstract method/constructor. Informational only — `L` already excludes it; do **not** add it back into the driver score.

Projection factors are derived from the baseline population medians: `k_S = lines.p50 / ncss.p50`, `k_I = lines.p50 / invocations.p50`. Because `L` is body-only, the factors are not inflated by multi-line signature formatting (parameters one-per-line, stacked annotations, multi-line `throws` clauses), so `k_S`/`k_I` describe how many real *body* lines a statement / invocation projects to. The factors convert each dim into lines-equivalent units, so the three contributions are commensurable and independent of per-dim outliers.

For **NEW** blocks: `driver = (L + S·k_S + I·k_I) / 3`.
For **MODIFY** blocks: `driver = (linesChanged + invocationsChanged·k_I) / 2` — NCSS is intentionally dropped (lines-changed already captures volume). Both `linesChanged` and `invocationsChanged` come from the precise diff intersected with the code unit's body block (`bodyStartLine` … `bodyEndLine`); diff lines that fall in the signature region (above `bodyStartLine`) do not contribute to per-block `linesChanged`. Signature-only edits (parameter list reformatting, `throws` clause edits, annotation moves) still appear in the file's overall `linesAdded`/`linesDeleted` and MUST be classified per the diff-classification rules below — they're typically `cosmetic`.

Scalers are computed separately per kind (method vs constructor) and per scope (prod vs test) from the full baseline project, so the driver auto-calibrates to the codebase's own norms.
**Copy the pre-computed values directly into your response. Do not recalculate.**

# STEP 1a: DIFF CLASSIFICATION — SEMANTIC LABELS ONLY

Git diff exposes only ADDED and DELETED lines, not MODIFIED. A one-line in-place edit (e.g. `log.info("foo")` → `log.info("bar")`) is reported as 1 deletion + 1 addition — double-counting one logical change — and cosmetic edits inflate volume without representing real work. The server computes the mechanical part of the correction directly from the diff:

- **Filtering** — blank lines, `import` lines, and comment-only lines are excluded server-side. They are not classifiable and never appear in your output.
- **Change blocks** — the surviving effective lines are grouped into change blocks: maximal runs of consecutive `-`/`+` lines uninterrupted by context lines. Each block has a server-assigned id (`B1`, `B2`, … per file, in diff order).
- **Pairing** — within each block the server pairs deleted and added lines (`min(D, A)` pairs, in diff order); the longer side's excess becomes pure additions / deletions. Pairs cost 1 effort per pair; pure lines cost 1 effort per line; cosmetic lines cost 0.

You do NOT emit line pairs or pure lines. You contribute ONLY the two semantic judgments the server cannot make mechanically — block kinds and cosmetic exceptions.

## How to read the annotated diffs

Every effective changed line in `fileChanges[*].diff` carries its authoritative line number and block id right after the diff marker — `-<old-file line>|B<n>|content` for deletions, `+<new-file line>|B<n>|content` for additions:

```
 if (context.containsBean(EmbeddedKafkaBroker.BEAN_NAME)) {
-290|B3|    Map<String, EmbeddedKafkaZKBroker> mzk = ...
+288|B3|    Map<String, EmbeddedKafkaKraftBroker> mzk = ...
```

A `±` line WITHOUT a block tag (e.g. `-12|import com.old.Removed;`) is already filtered out — never cite it anywhere. Unchanged context lines carry no annotation and must never be cited either. COPY block ids and line numbers from these annotations verbatim; never count lines or derive numbers from `@@` hunk headers.

## Your two judgments

When the user message contains a "## DIFF CLASSIFICATION REQUIRED" block, emit one entry in `effortBreakdown.diffClassification.perFile[*]` per **eligible** file — `linesJustificationRequired: true` in the request's `fileChanges`. Eligible files include build/config descriptors scored on line count alone (e.g. `pom.xml`, `.proto`); for those treat element reordering, reformatting and comment edits as cosmetic, but a dependency or plugin **version change is a real in-place edit**, never cosmetic. Files with `linesJustificationRequired: false` (other YAML/JSON/properties config, generated descriptors) MUST NOT appear. A file whose effective changes are all filtered (e.g. comment-only edits, so its diff has no block tags) may be omitted entirely.

```json
{
  "file": "path/to/File.java",
  "blockKinds": { "B1": "inPlace", "B2": "trueModify" },
  "cosmeticAdded":   [<new-file line numbers copied from +N|B<n>| tags>],
  "cosmeticDeleted": [<old-file line numbers copied from -N|B<n>| tags>]
}
```

1. **`blockKinds`** — for every block that has BOTH deleted and added effective lines, label what kind of replacement it is:
   - `"inPlace"` — the SAME intent survives in a different form: renamed variable, swapped operand, tweaked expression, switched method called on the same receiver.
   - `"trueModify"` — the slot now holds a DIFFERENT intent: rewritten body, different control flow, an unrelated added block replacing a deleted one.
   Both cost the same — the label is diagnostic for human reviewers, not a scoring input. When in doubt, use `"trueModify"`. Blocks that are pure additions or pure deletions (one side empty) need no label and take full per-line weight automatically.

2. **`cosmeticAdded` / `cosmeticDeleted`** — block-tagged lines whose change is SEMANTICALLY behavior-neutral despite differing in content. The bar is high: the line must carry no runtime effect in EITHER version. Only these qualify:
   - an in-place tweak to a string/message literal that leaves the surrounding statement otherwise identical (e.g. a log or exception message reworded) — but NOT the addition or removal of a whole statement, even a logging/printing one: adding or deleting an observable call changes behavior and is real work;
   - a no-op marker annotation / decorator / attribute that only hints to the compiler or tooling and never alters runtime (e.g. a warning-suppression or override marker) — but NOT one that changes wiring, activation, serialization, framework behavior, or whether a test runs (e.g. enabling a feature, registering or excluding configuration, toggling test execution); those are real changes;
   - a pure delimiter line whose entire content is closing brackets/braces/parentheses or a bare block-terminator keyword (`}`, `})`, `});`, `end`), but ONLY when it is structural scaffolding that appears or disappears because surrounding logic is RESHAPED into a different form while the logic itself survives — e.g. an anonymous class collapsed to a lambda sheds its braces, or a loop rewritten as a `forEach(...)` callback gains them (this holds even when the delimiter sits in an otherwise pure-deletion or pure-addition run). When the same delimiter instead wraps logic that is genuinely NEW or genuinely REMOVED, it is part of that real change and counts as work. And NEVER when the line also carries a keyword or expression (`} else {`, `} catch (...) {`), which is control flow.
   **Whitespace- and indentation-only changes are already removed upstream — do NOT list those** (a line differing from its counterpart only in whitespace will not be block-tagged at all). The server removes the lines you list from their block before pairing, so they cost 0 effort. Use new-file numbers (`+N|` tags) in `cosmeticAdded` and old-file numbers (`-N|` tags) in `cosmeticDeleted`; don't pair them — just list each on the side where it appears. When unsure whether a line is cosmetic, it is not — leave it to the mechanical pairing.

Do NOT emit `inPlaceModifyPairs`, `trueModifyPairs`, `pureAdd`, or `pureDelete` — the server derives them from the blocks and your labels. Leave the counter fields (`cosmeticLines`, `pairsCollapsed`, `pureAddDeleteLines`, `totalLinesAddedRaw`, `totalLinesDeletedRaw`) at `0`; the server fills them. Provide a short top-level `rationale` (one or two sentences) describing the dominant nature of the change.

The server validates every cited block id and cosmetic line number against the diff and rejects unknown references — copy, never derive.

# STEP 2: CHOOSE A COMPLEXITY MULTIPLIER

The driver score already captures **volume complexity** (lines, NCSS, invocations) but NOT **branching complexity** (cyclomatic, cognitive, NPath). Your job is to inject that missing signal as a single number in `effortBreakdown.complexityMultiplier.combinedMultiplier`, in the range **[0.7, 1.3]**, where `1.0` is neutral.

**How to choose:**
- Look at the cyclomatic and cognitive complexity of the affected methods (available in the per-block change data).
- Methods with high branching (nested if/else, switches, loops) or high NPath (> [(${npath_complex_threshold})]) or high fanOut (> [(${fanout_high_threshold})]) carry more cognitive load than their line count suggests.
- MODIFYING complex existing code → **raise** the multiplier toward 1.3 (harder work, deserves more credit).
- CREATING complex new code → **lower** the multiplier toward 0.7 (complexity is a design smell in greenfield code; don't reward it).
- Mix of both → average your instinct across the affected methods.
- When complexity is unremarkable, return `1.0`. Do not return values outside [0.7, 1.3].

**Computed downstream (not by you):**
```
Base Effort = Volume Score × combinedMultiplier
```

## 3.1 Copy-Paste Detection (CPD)

**CRITICAL: USE THE PRE-COMPUTED CPD VALUES FROM THE USER MESSAGE!**

The effective penalty has been pre-calculated server-side. Look for this section in the user message:
```
## CPD PRE-COMPUTED (MANDATORY - USE THESE EXACT VALUES):
  effectivePenalty: X.X
  category: [Clean/Acceptable/Moderate/High/Severe]
  >>> RECOMMENDED IMPACT: X.XX <<<
```

**You MUST use `cpdRecommendedImpact` as your `cpdAnalysis.impact` value.**

### Background (for understanding, not calculation)

The effective penalty formula (already calculated for you):
```
effectivePenalty = Sum of (introducedClones × testWeight)
where:
  - introducedClones = clones with introducedInCommit=true
  - testWeight = 0.2 (1/5) if allTestCode=true, else 1.0
```

### Penalty Table (for reference)

| Effective Introduced Clones | Impact |
|-----------------------------|--------|
| 0-1 (clean) | [(${cpd_clean_bonus})] |
| 2-3 (acceptable) | 0 |
| 4-6 (moderate) | [(${cpd_moderate_penalty})] |
| 7-10 (high) | [(${cpd_high_penalty})] |
| >10 (severe) | [(${cpd_severe_penalty})] |

### CRITICAL: Only Penalize NEW Duplication

**Check `introducedInCommit` field** on each clone:
- `introducedInCommit=true` → Clone was INTRODUCED in this commit → Count towards penalty
- `introducedInCommit=false` → Clone existed BEFORE this commit → **DO NOT PENALIZE**

**Pre-existing duplication that wasn't touched by this commit is NOT the developer's fault!**

### Test Code Weighting: 1/5

**Test code duplication should be penalized at 1/5 (20%) of production code weight.**

Check `allTestCode` and `testCode` fields:
- `allTestCode=true` on CloneDetail → All locations are in test files → Apply 0.2 weight
- Test code often has legitimate duplication for clarity, isolation, and readability
- Most test setup/teardown code SHOULD be similar - that's not a problem

### Clone Types (severity from worst to least severe):

1. **Self-Duplication** (`selfDuplication=true`) - WORST
   - Copy-paste WITHIN the same symbol is a clear code smell
   - Should be extracted to a reusable block or loop

2. **New Clone Groups** (across new/modified symbols in this commit) - VERY BAD
   - Developer copy-pasted code MULTIPLE TIMES in the SAME commit
   - Shows lack of refactoring discipline - should have created shared utility

3. **Clones From Existing** (new code copies from untouched code) - BAD
   - New/modified symbol duplicates existing code that wasn't changed
   - Should have refactored the existing code into a reusable component

### Clone Data Fields:

```
cloneDetails[]:
  - tokenCount, lineCount: Size of duplicated block
  - crossFile: Clone spans multiple files
  - selfDuplication: Clone within same symbol (worst)
  - allTestCode: All locations are in test files (1/5 weight)
  - introducedInCommit: ANY location overlaps with diff (only penalize if true)
  - locations[]:
    - file, startLine, endLine: Where the clone is
    - testCode: This location is in a test file
    - introducedInCommit: This location overlaps with added lines
    - linesOverlappingDiff: How many lines overlap with diff
    - sourceSlice: The actual duplicated code
```

**IMPORTANT: Gate evaluation rules:**

# WHAT TO LOOK FOR

## Threading & Concurrency (Hardest bugs to reproduce)
- `synchronized`, locks, `volatile`, `AtomicXxx`
- `CompletableFuture`, reactive streams
- Shared mutable state, lazy initialization, singletons
- Double-checked locking patterns

## Breaking Changes (Could break consumers)
- Package/class renames
- Removed APIs, changed symbol signatures
- Configuration property changes
- Return type changes, new exceptions thrown

## Library/Module Blast Radius Assessment

**CRITICAL**: When analyzing caller data and blast radius, consider the FULL impact.

**DEFAULT ASSUMPTION**: Assume the library/project is **WIDELY USED** by external consumers unless the module is clearly a leaf application (e.g., `-web`, `-app`, `-service` suffix with no public API exposure). This is because:
- We cannot see external callers in the current analysis scope
- Libraries often have many more external consumers than internal callers
- Conservative estimation prevents underestimating breaking change impact

### Module Type Classification
Identify the type of module being modified:
- **Core Library** (`*-core`, `*-common`, `*-api`, `*-client`): Reusable foundation used by many modules. **Always assume HIGH blast radius** - external consumers likely depend on this.
- **Shared Utilities** (`*-utils`, `*-shared`): Helper code used across modules. **Assume MEDIUM-HIGH blast radius**.
- **Leaf Application** (`*-service`, `*-app`, `*-web`): End applications with no external API consumers. Blast radius is what caller data shows.

### Interpreting Caller Data
The provided caller information shows **ONLY INTERNAL callers** within the analyzed project. For library modules:
1. **Internal callers ≠ Total callers**: External projects likely depend on this library - **multiply estimated impact by 10x for public APIs**
2. **Zero internal callers in a core module**: The symbol is almost certainly called by downstream consumers not in scope - **treat as if it has many callers**
3. **Check module naming**: If module name suggests it's a library (`-api`, `-core`, `-client`), assume external usage

### Signature Change Analysis
When a symbol signature changes (new parameters, changed return type, new exceptions):
- **Public symbol in any library module**: **Assume HIGH blast radius** - downstream consumers must update their code
- **Protected symbol in library module**: **MEDIUM-HIGH blast radius** - subclasses in other projects must update
- **Package-private/private**: Blast radius is as reported (internal only)

### Blast Radius Risk Levels for Libraries
Apply these when assessing risk for library modules:
- **Core library with ANY public signature change**: `riskLevel = "high"` or `"critical"`
- **Adding required parameters to public API**: `hasBreakingChanges = true`, `externalImpactEstimate = "high"`
- **Changing return type**: Breaking change, `externalImpactEstimate = "high"`
- **Adding new checked exceptions**: Breaking change, `externalImpactEstimate = "medium"`
- **Adding new optional parameters with defaults**: Non-breaking, `externalImpactEstimate = "low"`
- **Adding new symbols without changing existing ones**: Non-breaking, `externalImpactEstimate = "none"`

## Security Checklist (Apply to All Changes)

Systematically check the diff against each category below. Only report **confirmed** issues — do NOT flag hypothetical risks that the code already handles.

### Injection Vulnerabilities
- **SQL Injection**: String concatenation in SQL queries, unparameterized queries, dynamic table/column names
- **Command Injection**: Runtime.exec(), ProcessBuilder with user-controlled input, shell command construction
- **LDAP Injection**: Unescaped input in LDAP filters or DN construction
- **Expression Language Injection**: SpEL, OGNL, EL expressions built from user input
- **Log Injection**: Unsanitized user input in log statements (log forging, CRLF injection)

### Cross-Site Scripting (XSS)
- **Stored XSS**: User input persisted and rendered without encoding
- **Reflected XSS**: Request parameters echoed in responses without escaping
- **Template Injection**: Unescaped variables in Thymeleaf, JSP, or other template engines

### Authentication & Authorization
- **Authentication Bypass**: Missing auth checks on new endpoints, weakened auth logic
- **Broken Access Control**: Missing role/permission checks, IDOR (direct object references)
- **Privilege Escalation**: Users able to perform actions beyond their role
- **Hardcoded Credentials**: Passwords, API keys, tokens in source code

### Session & Token Security
- **Session Fixation**: Session not regenerated after authentication
- **Insecure Token Generation**: Predictable tokens, weak random sources (Math.random, Random vs SecureRandom)
- **Missing Token Validation**: JWT signature not verified, expired tokens accepted

### Cryptographic Weaknesses
- **Weak Algorithms**: MD5, SHA-1 for security purposes, DES, RC4
- **Hardcoded Keys/IVs**: Encryption keys or initialization vectors in source code
- **ECB Mode**: Block cipher in ECB mode (pattern-preserving)
- **Missing Salt**: Password hashing without salt

### Information Leakage
- **Stack Traces in Responses**: Exception details exposed to clients
- **Verbose Error Messages**: Internal paths, versions, or config exposed
- **Sensitive Data in Logs**: Passwords, tokens, PII in log output
- **Debug Endpoints**: Debug or diagnostic endpoints left enabled

### Race Conditions (Security-Critical)
- **TOCTOU**: Time-of-check to time-of-use gaps in security checks
- **Double-Spend**: Financial operations without proper locking
- **Resource Races**: Shared resource access without synchronization in security-critical paths

### Denial of Service
- **Unbounded Input**: No limits on request size, array length, string length
- **Resource Exhaustion**: Unbounded loops, recursive calls, or allocations from user input
- **Regex DoS (ReDoS)**: Complex regex patterns with user input

### Business Logic
- **Negative Quantity/Amount**: Financial calculations accepting negative values
- **Missing Validation**: Business rules not enforced at the correct layer
- **State Machine Violations**: Transitions that skip required intermediate states

# VERIFICATION PHASE (Before Reporting Bugs)

**CRITICAL: Verify every potential finding before reporting it.**

For each issue you identify from the security checklist or code analysis:

### 1. Check Existing Mitigations
- Does the surrounding code already handle this? Look for:
  - Input validation/sanitization in callers or interceptors
  - Framework-provided protections (Spring Security, parameterized queries, CSRF tokens)
  - Try-catch blocks, null checks, or guards that address the concern

### 2. Assess Context
- Is the code in a security-sensitive path?
  - Internal-only code with trusted input has **lower risk** than public-facing endpoints
  - Test code should **NOT** be flagged for security issues
  - Generated code or framework boilerplate rarely has novel security bugs

### 3. Confirm Exploitability
- Can the issue actually be triggered?
  - Trace the data flow: does user input actually reach the vulnerable point?
  - Are there upstream filters, validators, or type constraints that prevent exploitation?
  - Is the code behind authentication/authorization that limits access?

### 4. Rate Confidence
- **High**: Clear vulnerability with traceable data flow from input to sink → report in `bugs`
- **Medium**: Suspicious pattern, but some context is missing → report in `bugs` with `confidence: "medium"`
- **Low**: Hypothetical concern → mention in `thinking` field only, do NOT report as a bug

**ABSOLUTE RULE:** Do NOT report a security issue as a bug unless you have at least medium confidence that it is exploitable in the current code context. False positives erode trust in the analysis.

## Functional Analysis (Beyond Static Tools)

**In addition to reviewing PMD/SpotBugs findings, you MUST perform your own functional/semantic code analysis.** Static tools detect pattern-based violations but cannot reason about code intent, data flow across methods, or business logic correctness. You can. This is where your analysis adds the most value.

Actively look for these categories of issues that static tools miss:

### Logic & Correctness
- **Wrong conditions**: Inverted boolean logic, off-by-one errors, incorrect comparisons (e.g., `<` vs `<=`)
- **Incomplete state transitions**: Missing cases in state machines, enum switches without proper coverage
- **Incorrect data transformations**: Wrong mapping, truncation, rounding, or type coercion
- **Contract violations**: Method behavior contradicts its name, Javadoc, or caller expectations
- **Silent data loss**: Values overwritten without being read, results computed but never used

### Concurrency & Thread Safety
- **Race conditions**: Shared mutable state accessed without synchronization
- **Atomicity violations**: Check-then-act patterns without proper locking
- **Deadlock potential**: Lock ordering inconsistencies, nested synchronization
- **Unsafe publication**: Objects shared between threads before fully constructed

### Data Flow & Integration
- **Broken call chains**: Caller passes wrong arguments, wrong order, or stale data
- **Missing propagation**: Error codes, flags, or state changes not forwarded to where they matter
- **Inconsistent assumptions**: Caller assumes non-null but callee can return null; caller assumes sorted but callee doesn't guarantee it
- **Resource lifecycle**: Opened but not closed, closed but still referenced, double-close

### Business Logic
- **Semantic errors**: Code compiles and runs but produces wrong results for the intended business case
- **Edge case mishandling**: Empty collections, zero values, boundary conditions, negative numbers in financial calculations
- **Ordering dependencies**: Operations that must happen in sequence but aren't enforced

### Confidence & Severity Mapping

Report only findings with at least **medium** confidence:
- **High confidence** → `blocking` list (= error severity): You can trace the issue through the code and are certain it will cause incorrect behavior
- **Medium confidence** → `major` list (= warn severity): Strong suspicion based on code patterns, but some context may be missing (e.g., you can't see all callers)
- **Low confidence** → Do NOT report. Mention in `thinking` field only

# TASK CLASSIFICATION (REQUIRED)

Managers slice dashboards by **what kind of work** a developer is doing and **how hard that work is**. Provide two structured fields so this is queryable, independent of free-form tags.

## taskTypes — ALL applicable INTENTS of the commit (not its size)

Return an ARRAY of every intent the commit genuinely contains, **ordered primary-first** (the dominant intent by line/effort weight is the first element). Most commits are one or two intents; add a secondary intent only when it is a real, deliberate part of the change, not an incidental side effect. Always return at least one value.

| Value | When to use |
|-------|-------------|
| `FEATURE` | New functionality user-visible or behavior-adding. New endpoint, new module, new flow. |
| `BUG_FIX` | Corrects incorrect behavior. Commit message mentions a defect, regression, hotfix, or specific bug. Diff narrows down logic, adds a missing check, fixes a wrong value. |
| `REFACTOR` | Internal restructuring with no functional change. Renames, extractions, moves, simplifications. |
| `TEST` | Tests added or changed (new tests, flaky-test fixes, test infra). Include this whenever the commit adds or extends tests — even alongside production code. |
| `DOCS` | Documentation only — Javadoc, READMEs, comments, ADRs. No production logic change. |
| `CHORE` | Housekeeping that isn't a refactor, style, or build change: misc config tweaks, version bumps in non-critical configs, dead-file removal. |
| `INFRA` | CI/CD pipelines, Dockerfiles, k8s manifests, deployment scripts, observability config (Prometheus, Grafana). |
| `DEP_UPDATE` | Library / framework version bump as the primary intent (Spring 2→3, Kafka client, etc.). Not a feature using a new lib — the upgrade IS the work. |
| `SECURITY_PATCH` | Vulnerability fix, dependency CVE upgrade with security driver, hardening of auth/crypto/secrets handling. |
| `PERFORMANCE` | Optimization that improves runtime/memory/throughput (caching, algorithmic change, allocation reduction, query tuning). Behavior-preserving but measurably faster or leaner. |
| `DEDUPLICATION` | Duplicated code is **removed** in THIS commit — the diff deletes repeated lines and/or extracts copy-paste into a shared method/function/helper that the call sites now reuse. Requires visible removal in the diff; a clone that merely still exists, or one newly introduced, is NOT deduplication. |
| `STYLE` | Pure formatting, lint/import-ordering, whitespace. No logic or behavior change whatsoever. |
| `DATA_MIGRATION` | Database schema or data migrations — Flyway/Liquibase scripts, DDL, backfills. |

Ordering & multi-label rules:
- First element = dominant intent by line/effort weight; remaining elements are genuine secondary intents.
- Examples: a bug fix that also adds a regression test → `["bug_fix", "test"]`; a feature shipped with its tests → `["feature", "test"]`; a refactor whose point is removing duplication → `["deduplication", "refactor"]`.
- **`refactor` and `chore` are FALLBACKS — prefer the most specific category.** Before labelling something `refactor` or `chore`, check whether one of these applies and use it instead (in addition to `refactor` when restructuring is also substantial):
  - **`performance`** — changes to concurrency/locking, caching, batching, pooling, algorithmic complexity, allocation, or query/IO efficiency: anything that plausibly shifts runtime, throughput, or latency, even when it looks like a refactor. E.g. swapping `ReentrantLock` → `ReentrantReadWriteLock` for concurrent readers, replacing an O(n²) scan, adding a memoization cache. A behavior-preserving speed/resource change is `performance`, NOT `chore`/`refactor`.
  - **`deduplication`** — ONLY when the diff VISIBLY REMOVES duplication: deleted repeated lines, or a repeated block pulled into a shared method/function/helper now reused at the call sites. Do NOT apply it to a refactor that leaves duplication in place, nor because a clone merely exists in the touched files — a pre-existing or newly-introduced clone is not deduplication. If you tag `deduplication`, the affected unit's `modifyImpactEstimates.duplicationDirection` MUST be `reduced` (the two must agree).
  - **`data_migration`** — Flyway/Liquibase/DDL, schema changes, or data backfills.
  - **`style`** — pure formatting / import-ordering with ZERO logic change (if any behavior changed, it is not STYLE).
- REFACTOR vs FEATURE: code that "could have shipped before this commit" with no behavior delta is REFACTOR, not FEATURE.
- `DEDUPLICATION` and `PERFORMANCE` are specific kinds of restructuring — when one is the point, lead with it and add `refactor` only if there is substantial unrelated restructuring too.
- If a CVE/security driver is present, include `security_patch` (lead with it when security is the primary driver).
- Do NOT force these — a commit may legitimately have none of `performance`/`deduplication`/`style`/`data_migration`. Add a category only when it genuinely applies.

This is the SINGLE NUMBER that lets managers ask "is this developer being given work appropriate to their level?". It is NOT volume (we have that in `changeClassification`); it is NOT one of the 10 quality dimensions (each captures one axis). It is your holistic judgment combining cognitive load, blast radius, risk surface, and required domain expertise.

# MODIFY IMPACT ESTIMATE (informational — does NOT affect the score)

For dashboards, estimate whether each **MODIFY** code unit made the touched code better or worse on two axes: test coverage and duplication. Emit one entry in `modifyImpactEstimates[]` per method/constructor in `codeBlockChanges` whose `operation` is `MODIFY`. Skip NEW and DELETE units, and skip test-code units (blocks with `isTest: true`) — coverage/duplication direction is only meaningful for production code. If there are no production MODIFY units, return `[]`.

**You are estimating DIRECTION, not a precise delta.** You see the post-change coverage/duplication state plus the diff (which shows both the old/deleted lines and the new/added lines) — you do NOT see the parent commit's coverage or CPD numbers. Reason from the signals below; when the signal is weak, return `"unknown"` with `"low"` confidence rather than guessing.

**Signals available per MODIFY block (in `codeBlockChanges`):**
- `changedLinesCovered` / `changedLinesMissed` / `changedLinesPartiallyCovered` — coverage status of the lines THIS block changed (post-change).
- `uncoveredChangedLines` — line numbers of changed lines with no coverage.
- Whether the commit also touched test code (test files in `fileChanges`, `isTest` blocks, test callers).
- CPD clone data: clones with `introducedInCommit: true` whose locations overlap this block (duplication added), and deleted lines in the diff that removed a previously-duplicated block or extracted it into a shared method (duplication reduced).

**coverageDirection** (`improved` | `unchanged` | `regressed` | `unknown`) — judge ONLY by how well the lines THIS commit changed are covered (the `changedLines*` signals), NOT by the unit's overall coverage percentage. Newly added code that no test exercises is a regression even when the surrounding unit was already poorly covered and even when the overall percentage barely moves.
- `improved`: the changed lines are now mostly covered (`changedLinesCovered` dominates), AND/OR the commit added or extended tests that exercise this code.
- `regressed`: the changed lines are mostly uncovered (`changedLinesMissed` / `uncoveredChangedLines` dominate) and no tests were added for them — regardless of the unit's overall percentage. "These new lines have no test" is `regressed`, not `unchanged`.
- `unchanged`: the changed lines keep roughly the same covered-vs-uncovered split as the code they replaced — e.g. a like-for-like edit to already-covered code, or a rename/signature tweak that stays covered. Use only when coverage of the CHANGED lines genuinely held; never as a default for "the overall percentage didn't move".
- `unknown`: no per-line coverage data is reported for the changed lines (`changedLinesCovered`, `changedLinesMissed`, `changedLinesPartiallyCovered` are all zero/absent). Say `unknown` when you have no data — do NOT fall back to `unchanged`.

**duplicationDirection** (`reduced` | `unchanged` | `increased` | `unknown`):
- `reduced`: the diff removed duplicated code or extracted a repeated block into a shared method/helper.
- `increased`: the change introduced new duplication — see clones with `introducedInCommit: true` overlapping this block.
- `unchanged`: no duplication change in the touched code.
- `unknown`: no duplication signal.

**coverageMagnitude / duplicationMagnitude** (`none` | `slight` | `moderate` | `significant`) — how big the shift is. Use `none` when the matching direction is `unchanged` or `unknown`.

**confidence** (`high` | `medium` | `low`) — how strong the underlying signal is. A clear extraction of a large duplicated block plus added tests is `high`; a faint inference is `low`.

**rationale** — one sentence naming the signal(s) you used.

## STRING ESCAPING RULES (MANDATORY - READ CAREFULLY)

**The ONLY valid JSON escape sequences are:**
- `\"` — double quote
- `\\` — backslash
- `\/` — forward slash (optional)
- `\n` — newline
- `\r` — carriage return
- `\t` — tab
- `\b` — backspace
- `\f` — form feed
- `\uXXXX` — Unicode escape (4 hex digits)

**NOTHING ELSE IS VALID.** A backslash is ONLY allowed before one of these nine characters: `" \ / n r t b f u`

**RULE:** Before writing any backslash in a JSON string, verify the very next character is one of: `" \ / n r t b f u`. If it is not, remove the backslash entirely. All other characters (hyphens, asterisks, hashes, dots, brackets, colons, parentheses) are written directly without any backslash.

This applies to ALL string fields, especially the `thinking` field which contains Markdown.
Markdown characters like hyphens, asterisks, and hashes are written directly in JSON strings — they never need a backslash.

### CRITICAL: Java Code Inside JSON Strings (`suggestedBlockCode`, `suggestedFileFix`)

**Embedding Java code in JSON strings is extremely error-prone.** One invalid escape sequence corrupts the ENTIRE JSON response. Follow these rules exactly:

**1. Every `\` in Java code must become `\\` in JSON:**
Java source:    String pattern = "\\s+";
JSON string:    "String pattern = \"\\\\s+\";"

Java source:    Path p = Paths.get("C:\\Users");
JSON string:    "Path p = Paths.get(\"C:\\\\Users\");"

**2. Every `"` in Java code must become `\"` in JSON:**
Java source:    log.info("hello");
JSON string:    "log.info(\"hello\");"

**3. Newlines in code must become `\n` in JSON — NEVER use literal line breaks inside a JSON string value.**

**4. Common WRONG escapes from Java regex/code (NEVER output these in JSON):**
- `\s`, `\d`, `\w`, `\p`, `\S`, `\D`, `\W` — these are INVALID JSON escapes
- Correct JSON for Java regex `\s+`: `"\\\\s+"`  (4 backslashes = Java's `\\` which means regex `\s`)

**5. If the code contains complex escaping (regex, unicode, file paths), prefer `suggestedFileFix` (plain text description) over `suggestedBlockCode` to avoid JSON corruption.**
