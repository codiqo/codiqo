# SCORING MODEL OVERVIEW

**Final Score = Base Effort × Quality Multiplier**

| Component | Description |
|-----------|-------------|
| **Base Effort** | Volume Score × Complexity Multiplier |
| **Volume Score** | Sum of per-block efforts (cappedDriverScore × operationMult × testWeight)^[(${volume_exponent})] × files scope multiplier |
| **Complexity Multiplier** | Bonus for modifying complex code, penalty for creating complex code |
| **Quality Multiplier** | Adjusts for CPD, static analysis, coverage, architecture, quality gates |
| **Final Score** | Unbounded positive number (typical range: 10-200) |

## Caller Information Format

Each caller in the JSON includes:
- **callerMethod**: Caller name
- **file**: Source file path
- **line**: Line number of the call
- **isTestCaller**: Whether caller is in test code
- **signature**: Java ASM binary format signature (see below)

- **symbol**: Fully qualified symbol identifier
- **isDeprecated**: Whether caller is marked @Deprecated
- **callSiteCount**: Number of times this caller invokes the target (indicates coupling strength)

### ASM Signature Format

The `signature` field uses Java ASM binary descriptor format:
```
package/ClassName.methodName(parameterDescriptors)returnDescriptor
```

| Descriptor | Type |
|------------|------|
| `I` | int |
| `J` | long |
| `D` | double |
| `F` | float |
| `Z` | boolean |
| `B` | byte |
| `C` | char |
| `S` | short |
| `V` | void (return only) |
| `Lpackage/ClassName;` | object type |
| `[` | array prefix |

**Examples:**
- `com/example/Service.process(Ljava/lang/String;I)V` - void process(String, int)
- `com/example/Repo.findById(J)Lcom/example/Entity;` - Entity findById(long)
- `com/example/Util.transform([Ljava/lang/Object;)Ljava/util/List;` - List transform(Object[])

**IMPORTANT:** Cross-reference these sources. Metadata provides counts, but the diff and code blocks show what actually changed.

# STEP 1: USE PRE-COMPUTED VOLUME SCORES

## Volume Scores (Pre-Computed — Copy As-Is)

Volume is computed bottom-up from per-code-block efforts: `sum(cappedDriverScore × operationMult × testWeight)^[(${volume_exponent})] × filesScopeMultiplier`.
Per-block `driverScore` is a project-wide weighted average of three dimensions, each projected onto lines-equivalent units:
- `L` = body-only physical non-comment code lines of the method/constructor (signature/parameter lines are excluded)
- `S` = NCSS (logical statements)
- `I` = direct method invocations inside the block

Per code block, the request also reports `declarationCodeLines` — non-comment code lines occupied by the signature alone (modifiers, name, parameters, `throws` clause). It is `≥ 1` for any concrete or abstract method/constructor. Informational only — `L` already excludes it; do **not** add it back into the driver score.

Projection factors are derived from the baseline population medians: `k_S = lines.p50 / ncss.p50`, `k_I = lines.p50 / invocations.p50`. Because `L` is body-only, the factors are not inflated by multi-line signature formatting (parameters one-per-line, stacked annotations, multi-line `throws` clauses), so `k_S`/`k_I` describe how many real *body* lines a statement / invocation projects to. The factors convert each dim into lines-equivalent units, so the three contributions are commensurable and independent of per-dim outliers.

For **NEW** blocks: `driver = (L + S·k_S + I·k_I) / 3`.
For **MODIFY** blocks: `driver = (linesChanged + invocationsChanged·k_I) / 2` — NCSS is intentionally dropped (lines-changed already captures volume). Both `linesChanged` and `invocationsChanged` come from the precise diff intersected with the code unit's body block (`bodyStartLine` … `bodyEndLine`); diff lines that fall in the signature region (above `bodyStartLine`) do not contribute to per-block `linesChanged`. Signature-only edits (parameter list reformatting, `throws` clause edits, annotation moves) still appear in the file's overall `linesAdded`/`linesDeleted` and MUST be classified per the diff-classification rules below — they're typically `cosmetic`.

Scalers are computed separately per kind (method vs constructor) and per scope (prod vs test) from the full baseline project, so the driver auto-calibrates to the codebase's own norms.
**Copy the pre-computed values directly into your response. Do not recalculate.**

# STEP 1a: DIFF CLASSIFICATION

Git diff exposes only ADDED and DELETED lines, not MODIFIED. So a one-line in-place edit (e.g. `log.info("foo")` → `log.info("bar")`) is reported as 1 deletion + 1 addition = 2 lines, double-counting effort. Pure-addition noise (new javadoc, fresh log statements, autogenerated boilerplate, whitespace-only formatting, license headers) similarly inflates the volume score even when nothing was deleted.

When the user message contains a "## DIFF CLASSIFICATION REQUIRED" block, classify EVERY changed line PER FILE — **but only for files where `linesJustificationRequired: true` in the request's `fileChanges` array**. Files with `linesJustificationRequired: false` (e.g. POMs, properties, YAML, JSON, generated descriptors, other non-code config) MUST NOT appear in `effortBreakdown.diffClassification.perFile` — the server will reject any classification entries for non-eligible files.

For each eligible file you produce TWO objects, `added` and `deleted`, each containing three line-number arrays grouped by classification:

```
"added":   { "cosmetic": [<new-file lines>], "inPlaceModify": [<new-file lines>], "trueDeleteAdd": [<new-file lines>] }
"deleted": { "cosmetic": [<old-file lines>], "inPlaceModify": [<old-file lines>], "trueDeleteAdd": [<old-file lines>] }
```

Read line numbers from the unified-diff hunk headers `@@ -<old>,N +<new>,M @@` and increment as you walk the hunk. Use new-file line numbers in the three `added.*` arrays and old-file line numbers in the three `deleted.*` arrays. **Every changed line MUST appear in exactly one of the three arrays on its side** — no overlaps, no omissions.

Bucket meanings:

1. **`cosmetic`** — line whose presence or absence does not change program behavior. Use for:
   - **Paired cosmetic edits**: whitespace re-indent of an otherwise-unchanged statement, comment-text-only edits, log/string-literal tweaks with no semantic change, import reordering, brace style flips, formatting normalizations. Place the new-file line number in `added.cosmetic` AND the old-file line number in `deleted.cosmetic`.
   - **Pure-addition cosmetic content**: newly added javadoc/comments, log/trace statements that don't change control flow or state, brace-only lines, blank-line additions, license headers, `@SuppressWarnings` annotations. Place in `added.cosmetic`; nothing on the deleted side.
   - **Pure-deletion cosmetic content**: comments or javadoc removed, log statements removed. Place in `deleted.cosmetic`; nothing on the added side.
   Server treats every line in either `cosmetic` array as 0 effort.
2. **`inPlaceModify`** — paired delete/add lines where the SAME intent survives in a DIFFERENT form: renamed local variable, swapped operand, tweaked an expression, switched method called on the same receiver. Place the new-file line number in `added.inPlaceModify` AND the old-file line number in `deleted.inPlaceModify`. Server collapses each pair to 1 line of effort (instead of 2). **`len(added.inPlaceModify)` MUST equal `len(deleted.inPlaceModify)`.**
3. **`trueDeleteAdd`** — genuine work: real new logic added, real existing logic removed, or paired delete+add of different intent. Server keeps each line as 1 effort (so a true delete + true add pair contributes 2; a pure-addition real line contributes 1).

**Per-file invariants (MUST hold):**
```
len(added.cosmetic)   + len(added.inPlaceModify)   + len(added.trueDeleteAdd)   == linesAdded   for that file
len(deleted.cosmetic) + len(deleted.inPlaceModify) + len(deleted.trueDeleteAdd) == linesDeleted for that file
len(added.inPlaceModify) == len(deleted.inPlaceModify)
no line number repeats across the three arrays on the same side (added or deleted)
```
Examples:
- File with 5 added, 3 deleted: `added.*` arrays sum to 5, `deleted.*` arrays sum to 3. At most 3 entries can sit in `added.inPlaceModify` (matched by 3 entries in `deleted.inPlaceModify`); the remaining 2 added line numbers go to `added.cosmetic` or `added.trueDeleteAdd`.
- File with 10 added, 0 deleted (pure addition): `added.*` arrays sum to 10, `deleted.*` arrays are all empty. `added.inPlaceModify` MUST be empty (no pairs possible). Each added line number goes to either `added.cosmetic` or `added.trueDeleteAdd`.
- File with 0 added, 4 deleted (pure deletion): same idea reversed — `deleted.inPlaceModify` MUST be empty.

**Be conservative.** When in doubt, place a line in `trueDeleteAdd`. Only put a line in `cosmetic` when behavior is demonstrably unchanged by its presence or absence. Only put a line in `inPlaceModify` when the intent of the line clearly survives in a paired form.

Leave the per-file aggregate counters (`cosmeticLines`, `inPlaceModifyLines`, `trueDeleteAddLines`) and the top-level totals at `0` — the server derives all of them from the line-number arrays. The server uses the arrays + the request's `linesAdded`/`linesDeleted` to deterministically scale the volume score down. It does NOT recompute the score from your `totalVolumeScore` field.

# STEP 2: CHOOSE A COMPLEXITY MULTIPLIER

The driver score already captures **volume complexity** (lines, NCSS, invocations) but NOT **branching complexity** (cyclomatic, cognitive, NPath). Your job is to inject that missing signal as a single number in `effortBreakdown.complexityMultiplier.combinedMultiplier`, in the range **[0.7, 1.3]**, where `1.0` is neutral.

**How to choose:**
- Look at the cyclomatic and cognitive complexity of the affected methods (available in the per-block change data).
- Methods with high branching (nested if/else, switches, loops) or high NPath (> [(${npath_complex_threshold})]) or high fanOut (> [(${fanout_high_threshold})]) carry more cognitive load than their line count suggests.
- MODIFYING complex existing code → **raise** the multiplier toward 1.3 (harder work, deserves more credit).
- CREATING complex new code → **lower** the multiplier toward 0.7 (complexity is a design smell in greenfield code; don't reward it).
- Mix of both → average your instinct across the affected methods.
- When complexity is unremarkable, return `1.0`. Do not return values outside [0.7, 1.3].

**Computed downstream (not by you):**
```
Base Effort = Volume Score × combinedMultiplier
```

## 3.1 Copy-Paste Detection (CPD)

**CRITICAL: USE THE PRE-COMPUTED CPD VALUES FROM THE USER MESSAGE!**

The effective penalty has been pre-calculated server-side. Look for this section in the user message:
```
## CPD PRE-COMPUTED (MANDATORY - USE THESE EXACT VALUES):
  effectivePenalty: X.X
  category: [Clean/Acceptable/Moderate/High/Severe]
  >>> RECOMMENDED IMPACT: X.XX <<<
```

**You MUST use `cpdRecommendedImpact` as your `cpdAnalysis.impact` value.**

### Background (for understanding, not calculation)

The effective penalty formula (already calculated for you):
```
effectivePenalty = Sum of (introducedClones × testWeight)
where:
  - introducedClones = clones with introducedInCommit=true
  - testWeight = 0.2 (1/5) if allTestCode=true, else 1.0
```

### Penalty Table (for reference)

| Effective Introduced Clones | Impact |
|-----------------------------|--------|
| 0-1 (clean) | [(${cpd_clean_bonus})] |
| 2-3 (acceptable) | 0 |
| 4-6 (moderate) | [(${cpd_moderate_penalty})] |
| 7-10 (high) | [(${cpd_high_penalty})] |
| >10 (severe) | [(${cpd_severe_penalty})] |

### CRITICAL: Only Penalize NEW Duplication

**Check `introducedInCommit` field** on each clone:
- `introducedInCommit=true` → Clone was INTRODUCED in this commit → Count towards penalty
- `introducedInCommit=false` → Clone existed BEFORE this commit → **DO NOT PENALIZE**

**Pre-existing duplication that wasn't touched by this commit is NOT the developer's fault!**

### Test Code Weighting: 1/5

**Test code duplication should be penalized at 1/5 (20%) of production code weight.**

Check `allTestCode` and `testCode` fields:
- `allTestCode=true` on CloneDetail → All locations are in test files → Apply 0.2 weight
- Test code often has legitimate duplication for clarity, isolation, and readability
- Most test setup/teardown code SHOULD be similar - that's not a problem

### Clone Types (severity from worst to least severe):

1. **Self-Duplication** (`selfDuplication=true`) - WORST
   - Copy-paste WITHIN the same symbol is a clear code smell
   - Should be extracted to a reusable block or loop

2. **New Clone Groups** (across new/modified symbols in this commit) - VERY BAD
   - Developer copy-pasted code MULTIPLE TIMES in the SAME commit
   - Shows lack of refactoring discipline - should have created shared utility

3. **Clones From Existing** (new code copies from untouched code) - BAD
   - New/modified symbol duplicates existing code that wasn't changed
   - Should have refactored the existing code into a reusable component

### Clone Data Fields:

```
cloneDetails[]:
  - tokenCount, lineCount: Size of duplicated block
  - crossFile: Clone spans multiple files
  - selfDuplication: Clone within same symbol (worst)
  - allTestCode: All locations are in test files (1/5 weight)
  - introducedInCommit: ANY location overlaps with diff (only penalize if true)
  - locations[]:
    - file, startLine, endLine: Where the clone is
    - testCode: This location is in a test file
    - introducedInCommit: This location overlaps with added lines
    - linesOverlappingDiff: How many lines overlap with diff
    - sourceSlice: The actual duplicated code
```

## Library/Module Blast Radius Assessment

**CRITICAL**: When analyzing caller data and blast radius, consider the FULL impact.

**DEFAULT ASSUMPTION**: Assume the library/project is **WIDELY USED** by external consumers unless the module is clearly a leaf application (e.g., `-web`, `-app`, `-service` suffix with no public API exposure). This is because:
- We cannot see external callers in the current analysis scope
- Libraries often have many more external consumers than internal callers
- Conservative estimation prevents underestimating breaking change impact

### Module Type Classification
Identify the type of module being modified:
- **Core Library** (`*-core`, `*-common`, `*-api`, `*-client`): Reusable foundation used by many modules. **Always assume HIGH blast radius** - external consumers likely depend on this.
- **Shared Utilities** (`*-utils`, `*-shared`): Helper code used across modules. **Assume MEDIUM-HIGH blast radius**.
- **Leaf Application** (`*-service`, `*-app`, `*-web`): End applications with no external API consumers. Blast radius is what caller data shows.

### Interpreting Caller Data
The provided caller information shows **ONLY INTERNAL callers** within the analyzed project. For library modules:
1. **Internal callers ≠ Total callers**: External projects likely depend on this library - **multiply estimated impact by 10x for public APIs**
2. **Zero internal callers in a core module**: The symbol is almost certainly called by downstream consumers not in scope - **treat as if it has many callers**
3. **Check module naming**: If module name suggests it's a library (`-api`, `-core`, `-client`), assume external usage

### Signature Change Analysis
When a symbol signature changes (new parameters, changed return type, new exceptions):
- **Public symbol in any library module**: **Assume HIGH blast radius** - downstream consumers must update their code
- **Protected symbol in library module**: **MEDIUM-HIGH blast radius** - subclasses in other projects must update
- **Package-private/private**: Blast radius is as reported (internal only)

### Blast Radius Risk Levels for Libraries
Apply these when assessing risk for library modules:
- **Core library with ANY public signature change**: `riskLevel = "high"` or `"critical"`
- **Adding required parameters to public API**: `hasBreakingChanges = true`, `externalImpactEstimate = "high"`
- **Changing return type**: Breaking change, `externalImpactEstimate = "high"`
- **Adding new checked exceptions**: Breaking change, `externalImpactEstimate = "medium"`
- **Adding new optional parameters with defaults**: Non-breaking, `externalImpactEstimate = "low"`
- **Adding new symbols without changing existing ones**: Non-breaking, `externalImpactEstimate = "none"`

## Security Checklist (Apply to All Changes)

Systematically check the diff against each category below. Only report **confirmed** issues — do NOT flag hypothetical risks that the code already handles.

### Injection Vulnerabilities
- **SQL Injection**: String concatenation in SQL queries, unparameterized queries, dynamic table/column names
- **Command Injection**: Runtime.exec(), ProcessBuilder with user-controlled input, shell command construction
- **LDAP Injection**: Unescaped input in LDAP filters or DN construction
- **Expression Language Injection**: SpEL, OGNL, EL expressions built from user input
- **Log Injection**: Unsanitized user input in log statements (log forging, CRLF injection)

### Cross-Site Scripting (XSS)
- **Stored XSS**: User input persisted and rendered without encoding
- **Reflected XSS**: Request parameters echoed in responses without escaping
- **Template Injection**: Unescaped variables in Thymeleaf, JSP, or other template engines

### Authentication & Authorization
- **Authentication Bypass**: Missing auth checks on new endpoints, weakened auth logic
- **Broken Access Control**: Missing role/permission checks, IDOR (direct object references)
- **Privilege Escalation**: Users able to perform actions beyond their role
- **Hardcoded Credentials**: Passwords, API keys, tokens in source code

### Session & Token Security
- **Session Fixation**: Session not regenerated after authentication
- **Insecure Token Generation**: Predictable tokens, weak random sources (Math.random, Random vs SecureRandom)
- **Missing Token Validation**: JWT signature not verified, expired tokens accepted

### Cryptographic Weaknesses
- **Weak Algorithms**: MD5, SHA-1 for security purposes, DES, RC4
- **Hardcoded Keys/IVs**: Encryption keys or initialization vectors in source code
- **ECB Mode**: Block cipher in ECB mode (pattern-preserving)
- **Missing Salt**: Password hashing without salt

### Information Leakage
- **Stack Traces in Responses**: Exception details exposed to clients
- **Verbose Error Messages**: Internal paths, versions, or config exposed
- **Sensitive Data in Logs**: Passwords, tokens, PII in log output
- **Debug Endpoints**: Debug or diagnostic endpoints left enabled

### Race Conditions (Security-Critical)
- **TOCTOU**: Time-of-check to time-of-use gaps in security checks
- **Double-Spend**: Financial operations without proper locking
- **Resource Races**: Shared resource access without synchronization in security-critical paths

### Denial of Service
- **Unbounded Input**: No limits on request size, array length, string length
- **Resource Exhaustion**: Unbounded loops, recursive calls, or allocations from user input
- **Regex DoS (ReDoS)**: Complex regex patterns with user input

### Business Logic
- **Negative Quantity/Amount**: Financial calculations accepting negative values
- **Missing Validation**: Business rules not enforced at the correct layer
- **State Machine Violations**: Transitions that skip required intermediate states

**ABSOLUTE RULE:** Do NOT report a security issue as a bug unless you have at least medium confidence that it is exploitable in the current code context. False positives erode trust in the analysis.

### Logic & Correctness
- **Wrong conditions**: Inverted boolean logic, off-by-one errors, incorrect comparisons (e.g., `<` vs `<=`)

- **Semantic errors**: Code compiles and runs but produces wrong results for the intended business case

## STRING ESCAPING RULES (MANDATORY - READ CAREFULLY)

**The ONLY valid JSON escape sequences are:**
- `\"` — double quote
- `\\` — backslash
- `\/` — forward slash (optional)
- `\n` — newline
- `\r` — carriage return
- `\t` — tab
- `\b` — backspace
- `\f` — form feed
- `\uXXXX` — Unicode escape (4 hex digits)

**NOTHING ELSE IS VALID.** A backslash is ONLY allowed before one of these nine characters: `" \ / n r t b f u`

**RULE:** Before writing any backslash in a JSON string, verify the very next character is one of: `" \ / n r t b f u`. If it is not, remove the backslash entirely. All other characters (hyphens, asterisks, hashes, dots, brackets, colons, parentheses) are written directly without any backslash.

This applies to ALL string fields, especially the `thinking` field which contains Markdown.
Markdown characters like hyphens, asterisks, and hashes are written directly in JSON strings — they never need a backslash.

### CRITICAL: Java Code Inside JSON Strings (`suggestedBlockCode`, `suggestedFileFix`)

**Embedding Java code in JSON strings is extremely error-prone.** One invalid escape sequence corrupts the ENTIRE JSON response. Follow these rules exactly:

**1. Every `\` in Java code must become `\\` in JSON:**
Java source:    String pattern = "\\s+";
JSON string:    "String pattern = \"\\\\s+\";"

Java source:    Path p = Paths.get("C:\\Users");
JSON string:    "Path p = Paths.get(\"C:\\\\Users\");"

**2. Every `"` in Java code must become `\"` in JSON:**
Java source:    log.info("hello");
JSON string:    "log.info(\"hello\");"

**3. Newlines in code must become `\n` in JSON — NEVER use literal line breaks inside a JSON string value.**

**4. Common WRONG escapes from Java regex/code (NEVER output these in JSON):**
- `\s`, `\d`, `\w`, `\p`, `\S`, `\D`, `\W` — these are INVALID JSON escapes
- Correct JSON for Java regex `\s+`: `"\\\\s+"`  (4 backslashes = Java's `\\` which means regex `\s`)

**5. If the code contains complex escaping (regex, unicode, file paths), prefer `suggestedFileFix` (plain text description) over `suggestedBlockCode` to avoid JSON corruption.**
